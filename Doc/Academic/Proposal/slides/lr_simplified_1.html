01 Zero-Knowledge Authentication

Reference Focus & Problem Statement Technical Approach Findings Strength / Weakness Future Direction
Bhattacharya, S., et al. (2024). Enhancing digital privacy: The application of zero-knowledge proofs in authentication systems.
Focus: ZKPs (SNARKs vs STARKs).

Problem: Centralized password databases create single points of failure. 		zk-SNARKs & zk-STARKs ZKPs verify users without data transmission. STARKs avoid "trusted setup" but have larger proof sizes. Strength Distinguishes zk-SNARK vs zk-STARK. Weakness High complexity barriers. Developing user-friendly frameworks.
Chen, Z., et al. (2023). A survey on zero-knowledge authentication for Internet of Things. Electronics.
Focus: ZKP on weak devices (Mobile/IoT).

Problem: Standard ZKP is too heavy for devices with low battery. 		Lightweight ZKP (Groth16) Non-interactive ZKP and zk-SNARKs (Groth16) are best for low-power environments. Strength Validates lightweight ZKP. Weakness Focuses mainly on sensors. Optimizing the "Setup Phase".
Luong, D. A., & Park, J. H. (2023). Privacy-preserving identity management system on blockchain using zk-SNARK.
Focus: Self-Sovereign Identity.

Problem: "Log in with Google" tracks user activity; users lose privacy. 		Blockchain + zk-SNARKs Verified users without storing passwords. Computation was heavy on client side. Strength Practical implementation details. Weakness Blockchain fees/cost. Reducing proof generation cost.
[Authored Group]. (2025). A secure framework for privacy-preserving analytics in healthcare records.
Focus: ZKP for sensitive records.

Problem: Privacy laws (GDPR) make sharing data for analysis difficult. 		Privacy-Preserving Framework ZKP allows "Auditability without Visibility"—checking math without seeing data. Strength High-stakes trust model. Weakness Complex multi-party setup. Integration with Federated Learning.

02 End-to-End Encryption (E2EE) Protocols

Reference Focus & Problem Statement Technical Approach Findings Strength / Weakness Future Direction
Alatawi, M., & Saxena, N. (2023). SoK: An analysis of end-to-end encryption and authentication ceremonies.
Focus: Signal Protocol & Key Verification.

Problem: Encryption works, but users fail to verify keys (QR codes) manually. 		Signal Protocol & Double Ratchet Double Ratchet Algorithm is robust, but "Authentication Ceremonies" are the weak link. Strength Identifies human error in E2EE. Weakness No technical fix proposed. Automated authentication ceremonies.
Ilesanmi, M. (2025). Threat modeling for enterprise secure messaging applications. Journal of Cybersecurity.
Focus: Attack surfaces in Slack/Teams.

Problem: Metadata (who talks to whom) is often not encrypted, even if messages are. 		STRIDE Threat Modeling E2EE protects transit, but endpoint security and metadata remain vulnerable. Strength Specific to chat domain. Weakness Theoretical modeling only. Minimizing metadata collection.
Barnes, R., et al. (2023). The Messaging Layer Security (MLS) Protocol (RFC 9420). IETF Standards Track.
Focus: Scaling E2EE for Groups.

Problem: Signal Protocol (Double Ratchet) is inefficient (slow) for large group chats. 		TreeKEM (Group Key Agreement) Uses binary trees to manage group keys, reducing cost from linear O(n) to logarithmic O(log n). Strength High Scalability. Weakness New/Complex to implement. Interoperability (MIMI IETF).
[Research Group]. (2025). Implementation of Secure E2EE Chat Application in Microservice Architecture. MDPI.
Focus: Full Stack Implementation.

Problem: Integrating encryption keys securely between client and server APIs. 		AES-256 & Diffie-Hellman Successfully integrated DH Key Exchange for shared secrets and AES-256 for message payload encryption. Strength Practical Dev Guide. Weakness Standard architecture (not novel). Adding Post-Quantum layers.

03 Current Machine-Learning Anomaly Detection (On-Device)

Reference Focus & Problem Statement Technical Approach Findings Strength / Weakness Future Direction
Srinivasan, K., et al. (2025). Insider threat detection and secure data transfer leveraging bidirectional LSTM.
Focus: Insider Threat Detection.

Problem: Standard rules miss subtle behavioral changes in authorized users. 		Bi-Directional LSTM (Deep Learning) Used Bi-LSTM to understand "context" of user actions, achieving higher accuracy than static rules. Strength High accuracy on time-series. Weakness Computationally heavy. optimizing for edge devices.
Gaurav, A., et al. (2025). AI-driven smishing detection in Android devices using TinyBERT.
Focus: On-Device Smishing (SMS Phishing).

Problem: Standard BERT models are too big for phones. 		TinyBERT (Lightweight AI) TinyBERT achieved 96.8% accuracy on Android with low battery impact. Strength Tailored for Android OS. Weakness Limited to text analysis. Real-time prevention systems.
Bispo, G. D., et al. (2025). PHILDER: Lightweight framework for intelligent phishing detection on resource-limited devices.
Focus: Edge Computing Detection.

Problem: Cloud detection causes latency and privacy risks (URL leakage). 		Edge AI (ARM Architecture) Proved that Edge AI can detect threats in <100ms without sending data to the cloud. Strength Validates local privacy. Weakness Limited to URL detection. Expanding to QR codes.
Natha, S. (2022). A systematic review of anomaly detection using machine and deep learning techniques.
Focus: ML vs Deep Learning.

Problem: Static algorithms (like SVM) fail when anomalies change nature. 		Deep Learning vs. SVM Deep Learning outperforms ML on high-dimensional data but requires more resources. Strength Critiques One-Class SVM. Weakness High computational cost. Resource-constrained ML.
