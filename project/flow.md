1. System Data Flow
This section describes how data moves through the system, specifically focusing on the unique ZKP Authentication and Mobile-to-Web handoff.

A. Authentication Flow (ZKP)
Mobile App Initiation:

User opens the mobile app and clicks "Login" or "Register".
Mobile App sends a POST /generate-mobile-access-token request to the Backend with deviceId and action.
Backend generates a Mobile Access Token (MAT) (valid for 5 mins), stores it, and returns it to the Mobile App.
Secure Web Handoff:

Mobile App constructs a URL (e.g., https://sentrizk.web/login?mat=<MAT>) and opens it in a WebView or external browser.
Web Frontend loads and immediately validates the MAT with the Backend.
If valid, the Backend allows the page to render; otherwise, access is denied.
Zero-Knowledge Proof Generation (Client-Side):

User enters credentials (or selects Wallet) on the Web Frontend.
Web Frontend retrieves public inputs:
Registration: Generates a random salt and secret.
Login: Fetches salt and nonce from Backend (GET /user/:username, GET /nonce).
Web Frontend (using snarkjs & circomlibjs) generates a ZKP Proof locally.
Circuit: commitment = Poseidon(secret, salt, unameHash)
Web Frontend sends the Proof and Public Signals (Commitment, Nullifier) to the Backend.
Verification & Session Creation:

Backend verifies the proof using the Verification Key and snarkjs.
Registration: Stores the commitment in the database.
Login: Verifies the commitment matches the stored one and checks the nonce to prevent replay attacks.
Backend issues a Session Token (JWT or random string) and returns it to the Web Frontend.
Return to Mobile:

Web Frontend redirects back to the Mobile App (via Deep Link) with the Session Token.
Mobile App stores the token securely (FlutterSecureStorage) for future API calls.
B. Chat Data Flow
Sending Message:

User types a message.
Mobile App (AI Module) runs TensorFlow Lite (Isolation Forest) on the typing pattern/content to detect anomalies.
If safe, Mobile App encrypts the message and sends it to Firestore (/chats/{receiverId}/messages).
Mobile App optionally uploads media to Firebase Storage.
Receiving Message:

Receiver App listens to Firestore updates.
Receiver App downloads the message, decrypts it, and stores it in Local Storage (Isar/Hive).
Receiver App triggers a delete operation on Firestore to ensure ephemeral storage (privacy).
2. Use Cases (Functional Requirements)
These describe the intended actions users perform within the system.

ID	Use Case	Actor	Description
UC-01	Register User	User	User creates an account using ZKP. A unique identity commitment is generated and stored on the server. No password is ever sent.
UC-02	Login User	User	User authenticates by proving knowledge of their secret and salt without revealing them. Requires a fresh nonce from the server.
UC-03	Generate MAT	Mobile App	App requests a temporary token to securely open the web authentication interface.
UC-04	Send Message	User	User sends an encrypted text or media message to another user.
UC-05	Receive Message	User	User receives messages in real-time via Firestore listeners and Push Notifications (FCM).
UC-06	Detect Anomaly	System (AI)	On-device AI analyzes user behavior (typing speed, frequency) to flag potential account compromise.
UC-07	Logout	User	User terminates the session. The backend invalidates the session token.
UC-08	View Chat History	User	User views locally stored messages (offline capable).
3. Misuse Cases (Security Threats)
These describe how an attacker might try to abuse the system and how SentriZK mitigates them.

ID	Misuse Case	Attacker Goal	Mitigation Strategy
MC-01	Replay Attack	Reuse a captured valid login proof to impersonate a user.	Nonce System: The backend issues a random nonce for every login attempt. The ZKP circuit binds the proof to this specific nonce. Old proofs fail verification.
MC-02	Man-in-the-Middle (Credential Theft)	Intercept network traffic to steal the user's password.	Zero-Knowledge Proofs: The password/secret is never sent over the network. Only a mathematical proof is transmitted. Intercepting the proof is useless without the secret inputs.
MC-03	Unauthorized Web Access	Open the registration/login page directly in a browser to bypass mobile security checks.	Mobile Access Token (MAT): The web pages require a valid, single-use MAT generated by the mobile app. Direct access without a MAT is rejected.
MC-04	Database Breach	Steal the user database to crack passwords.	Commitment Storage: The database only stores Poseidon hashes (commitments). These are mathematically impossible to reverse to find the original secret.
MC-05	Brute Force Attack	Repeatedly guess credentials or flood the server with proofs.	Rate Limiting: The backend enforces a strict limit (e.g., 10 requests/min) on /login and /register endpoints.
MC-06	Session Hijacking	Steal a session token to access the chat.	Short-Lived Sessions & AI: Sessions expire after 30 mins. Additionally, if the hijacker's behavior differs from the owner, the On-Device AI will flag the anomaly.
